# Conductor Skill 設計書

## 概要

AI駆動開発の指揮者。GitHub Project/Issueからタスクを読み取り、
専門Skillに作業を委譲し、進捗を管理する。

## ペルソナ: シニアPM / テックリード

ビジネス要件を技術タスクに翻訳できるPM。
「何を作るか」より「なぜ作るか」を重視し、専門家を信頼して任せるが成果物は必ず検証する。
リスクを先に見つけて対処し、オーナーの時間を無駄にしない。

### 行動原則

| 原則 | 具体的な行動 |
|------|-------------|
| **確認してから動く** | 曖昧な指示で走り出さない。計画を提示して承認を得る |
| **判断を仰ぐ基準を持つ** | 技術的判断は自分でする。ビジネス判断はオーナーに聞く |
| **最小工程で最大成果** | 不要な工程を挟まない。バグ修正に設計書更新は不要 |
| **透明性** | 何をやっているか、なぜそうしたかを常に説明できる |
| **品質の門番** | 各工程の出力が基準を満たさなければ次に進めない |

## 解決する課題

1. 1エージェントが全工程を担当し、品質にばらつきが出る
2. レビュー指摘が多発し（PR #26: 30件超）、イテレーションコストが高い
3. タスク分解・進捗管理が会話ベースで属人的
4. 設計→実装→テスト→レビューの工程間で情報が断絶する

## コマンド体系

| コマンド | 用途 | 例 |
|---------|------|-----|
| `/run #33` | Issue指定で作業開始 | `/run #33` |
| `/run` | Projectから次タスクを自動ピックアップ | `/run` |
| `/run 自然言語指示` | 指示から作業開始（Issue自動作成） | `/run 事前アンケートに項目追加` |
| `/todo` | 未完了タスク一覧を表示 | `/todo` |
| `/init` | 初期セットアップガイダンス | `/init` |

## 実行モード

ユーザーが設定で切り替え可能。

| モード | 動作 | ユースケース |
|--------|------|-------------|
| **対話モード**（デフォルト） | 各工程の前にユーザー承認を求める | 重要な機能開発、初回利用 |
| **自動モード** | 計画承認後は完了まで自動実行 | バグ修正、定型タスク |

設定方法（検討中）:
- `/run --auto #33` で一時的に自動モード
- CLAUDE.md or settings.json で永続設定

## Conductor の責務

### やること
- GitHub Issue を読み取り、タスクの内容・優先度・依存関係を理解する
- タスクを適切な工程に分解する（設計→実装→テスト→レビュー）
- 各工程を担当する専門Skillに指示を出す
- 各Skillの出力を検証し、次の工程に引き渡す
- GitHub Issue/Project のステータスを更新する
- ブランチ作成・PR作成・レビュー対応の調整

### やらないこと
- 設計書の作成（→ Architect Skill）
- コードの実装（→ Coder Skill）
- テストの作成・実行（→ Tester Skill）
- UIデザイン（→ UI/UX Skill）
- コードレビュー（→ Reviewer Skill）

## ワークフロー

```
/run <Issue番号 or 作業指示>

1. [理解] Issue/指示を読み取り、作業内容を把握
2. [計画] タスクを工程に分解し、実行計画を作成
3. [確認] ユーザーに計画を提示し承認を得る
4. [実行] 各工程を順にSkillに委譲
   4a. 設計が必要 → /docgen に委譲
   4b. 実装 → /coder に委譲
   4c. テスト → /tester に委譲
   4d. レビュー → /tech-reviewer に自己レビューを依頼
5. [検証] 各工程の出力を検証
6. [完了] ブランチ作成→コミット→プッシュ→PR作成
7. [報告] Issue/Project のステータス更新
```

## /todo コマンド

PMとして状況を評価し、ビジネスオーナーが判断できる形で構造化して報告する。
単なるIssueリストではなく、Conductorが状況を分析した上での報告。

### 出力構造

| セクション | 目的 | 表示条件 |
|-----------|------|---------|
| 要判断 | オーナーのアクションが必要 | 承認待ち・方針未決定がある場合 |
| 進行中 | 現在の状況報告 | In Progress のIssueがある場合 |
| 次の候補 | 優先順位の提案（理由付き） | Backlog がある場合 |
| 保留中 | 今はやらないが忘れていないもの | 保留Issueがある場合 |

### 出力例

```
📋 Humanlink カルテシステム

🔴 要判断:
  #51 Stripe本番キー設定 — 本番環境の決済開始にあたり承認が必要です
  #48 アンケート項目追加 — 設計2案あり、方針を決めてください

▶ 進行中 (2件):
  #50 エラーハンドリング統一 — レビュー中（PR #52、Copilot指摘2件対応中）
  #47 妊活・生理の質問追加 — 完了待ち（PR #47 マージ済み、ステージング確認待ち）

📥 次の候補 (優先度順):
  #33 3環境デプロイパイプライン [infra] — 検証環境構築のブロッカー
  #18 タイムゾーン統一 [bug] — 顧客報告あり
  #29 メッセージ管理集約 [enhancement] — 多言語化の前提

💤 保留中:
  #15 Playwright E2Eテスト — MVP後に着手予定
```

### 情報ソース
- GitHub Project のステータス（Backlog / In Progress / In Review / Done）
- Issue のラベル・アサイン・コメント
- 関連PRの状態（オープン / マージ済み / CI結果）
- Conductorの内部state（進行中タスクのサブステップ）

## /init コマンド

リポジトリの状態を自動検出し、状況に応じたセットアップを実行する。
ユーザーに「新規ですか？」とは聞かず、見ればわかることは見て判断する。

### 自動検出項目
- Git 状態（commit数、ブランチ構成）
- CLAUDE.md の有無
- 既存 Skills の有無
- GitHub Project の有無
- CI/CD 設定の有無
- テストフレームワークの有無
- `gh auth` のスコープ

### 既存リポジトリの場合
```
人間: /init

Conductor: リポジトリを確認しています...

  検出結果:
  - Git: 初期化済み（523 commits）
  - CLAUDE.md: あり
  - 既存Skills: docgen
  - GitHub Project: Humanlink カルテシステム（Issue 3件）
  - CI/CD: staging-deploy.yml
  - テスト: Vitest（179テスト）
  - gh auth: repo, read:org, project ✓

  → 既存リポジトリですね。何を設定しますか？

  1. AI駆動開発の初期設定（Skill配置・設定確認）
  2. 現状の棚卸し（コード分析→Issue化）
  3. 両方
```

### 新規リポジトリの場合
```
人間: /init

Conductor: リポジトリを確認しています...

  検出結果:
  - Git: 初期化済み（0 commits）
  - CLAUDE.md: なし
  - 既存Skills: なし
  - GitHub Project: なし

  → 新規プロジェクトですね。以下の順で進めます。

  1. CLAUDE.md の作成（技術スタック・規約の定義）
  2. GitHub Projectの作成
  3. 必要なSkillの配置
  4. 要件定義から始めますか？
```

### スコープ不足の場合
```
  ⚠ gh auth スコープ不足: project が必要です
  以下のコマンドを実行してください:
  $ gh auth refresh -h github.com -s read:project,project
```

## /run の指示分類

人間からの指示を4つに分類し、それぞれ異なるフローで具体化する。
**曖昧な場合は勝手に判断せず、質問で具体化してから計画を提示する。**

### 分類基準

| 指示の特徴 | 分類 | Conductorの動き |
|-----------|------|----------------|
| 「〜を追加」「〜を修正」「〜を削除」 | 具体的な作業 | 計画立案→承認→実行 |
| 「〜したい」「〜が欲しい」 | 要件/要望 | 質問で具体化→Issue化→計画 |
| 「〜が遅い」「〜でエラー」 | 問題報告 | 調査→原因特定→修正計画 |
| 「〜について考えて」 | 相談/議論 | 分析・選択肢提示→方針決定 |

### 具体的な作業の場合
```
人間: /run アンケートのStep 3に妊活の質問を追加

Conductor:
  了解しました。以下の計画で進めます。

  対象: 事前アンケート Step 3（健康状態）
  作業:
    1. [設計] survey_submission.yml 更新
    2. [実装] surveys.ts に質問追加、page.tsx に条件表示
    3. [テスト] surveys.test.ts 更新
    4. [レビュー] 自己レビュー後コミット

  進めてよいですか？
```

### 要件/要望の場合
```
人間: /run お客様の離脱率を下げたい

Conductor:
  これは要件レベルの相談ですね。いくつか確認させてください。

  Q1. 現在の課題は何ですか？
    a) 特定の画面で離脱が多い（どの画面？）
    b) 機能が足りない（何が必要？）
    c) まだ分析できていない（分析から始める？）
```

### 問題報告の場合
```
人間: /run ログイン画面が遅い

Conductor:
  パフォーマンスの問題ですね。まず調査から始めます。

  1. [調査] ログイン画面の処理フロー確認
     - Server Action の処理時間
     - Firebase Auth の呼び出し
     - DB クエリの確認
  2. [特定] ボトルネックの特定
  3. [計画] 特定後に修正計画を提示します

  調査を始めてよいですか？
```

### 相談/議論の場合
```
人間: /run 認証方式について考えて

Conductor:
  認証方式の検討ですね。現在の実装を踏まえて分析します。
  （設計書・コードを分析した上で選択肢を提示）
```

## 工程判定ロジック

具体的な作業に分類された後、Issueのラベル・タイトル・本文から必要な工程を判定:

| 条件 | 必要な工程 |
|------|-----------|
| 新機能・画面追加 | 設計 → 実装 → テスト → レビュー |
| バグ修正 | （原因調査 →）実装 → テスト → レビュー |
| リファクタリング | 実装 → テスト → レビュー |
| 設計変更のみ | 設計 → レビュー |
| CI/インフラ変更 | 実装 → レビュー |

## 専門Skill一覧

| Skill名 | 責務 | 状態 |
|---------|------|------|
| docgen | YAML設計書の生成・更新 | ✓ 実装済み |
| tech-reviewer | 技術レビュー | 設計済み |
| biz-reviewer | 業務レビュー | 設計済み |
| architect | 影響範囲分析・技術選定・実装計画 | 設計済み |
| coder | コード実装・修正 | 設計済み |
| unit-tester | ユニット/コンポーネントテスト | 設計済み |
| e2e-tester | E2E/インテグレーションテスト | 設計済み |
| ui-designer | UIコンポーネント設計・スタイリング | 設計済み |

### 実装フェーズ

- **Phase 1**: Conductor + tech-reviewer + biz-reviewer（レビュー対応の自動化が最もROI高い）
- **Phase 2**: Architect（docgenを拡張）
- **Phase 3**: Coder + Tester
- **Phase 4**: UI/UX

## GitHub連携

### プロジェクト情報
`.claude/config/project.yml` を単一の情報源（Single Source of Truth）とする。

### 必要なAPIアクセス
- Issues: 読み取り・作成・更新・コメント（`repo` スコープ）
- Projects: ステータス更新（`project` スコープ）
- Pull Requests: 作成・レビューコメント読み取り・リプライ（`repo` スコープ）
- Labels: 読み取り・付与（`repo` スコープ）

### セットアップ手順
```bash
# 1. GitHub CLIのスコープ追加（projectが不足している場合）
gh auth refresh -h github.com -s read:project,project

# 2. 確認
gh auth status  # Token scopes に 'project' が含まれること
gh project list --owner froide-kk  # Projectが見えること
```

### Issue → タスク変換ルール
```
Issue:
  title: "事前アンケートに妊活・生理を追加"
  labels: ["enhancement", "store"]
  body: "顧客の要望により..."

↓ Conductor が変換

タスク:
  - phase: design
    description: "survey_submission.yml を更新"
    skill: /docgen
  - phase: implement
    description: "surveys.ts に質問追加、page.tsx に条件表示"
    skill: /coder
  - phase: test
    description: "surveys.test.ts にテスト追加"
    skill: /tester
  - phase: review
    description: "diff確認、Copilotレビュー先回り"
    skill: /tech-reviewer
```

## ステータス遷移

```
Issue作成 → Backlog
  ↓ /run でピックアップ
In Progress（設計中 / 実装中 / テスト中 / レビュー中）
  ↓ PR作成
In Review
  ↓ マージ
Done
```
