# Skill: test-design-principles

## 目的

テストを書く際の「何を・どう・どの順で」を決める普遍的な原則。
特定のテストフレームワークの記法は別 Skill（`test-vitest`, `test-playwright` 等）に委ねる。

## 設計原則

### 1. 異常系を先に書く

正常系より先に異常系をテストする:

- バリデーションエラー
- 権限不足
- データ不存在
- 不正なフォーマット
- 必須フィールド欠落

理由: 異常系の方がバグが潜みやすく、正常系を後回しにしても全体の品質が下がりにくい。

### 2. 境界値を攻める

数値・文字列・配列の境界を必ずテスト:

- 最大値・最小値
- 0 / -1 / 1
- 空文字 / null / undefined
- 配列が空 / 1要素 / 多要素
- 文字数制限の境界（N-1, N, N+1）

### 3. 対称性を保つ

同じ性質のフィールドが複数ある場合、すべてに対して同じテストを書く:

- 姓・名どちらも空文字テストを書く（片方だけにしない）
- メール・電話どちらにも形式バリデーションテストを書く

### 4. テスト名で仕様を表現

テスト名だけ読めば仕様が分かるように書く:

- ❌ `test login`
- ✅ `returns 401 when password is incorrect`
- ✅ `creates user with valid input and emits event`

### 5. 独立性を保つ

各テストが単独で実行可能であること:

- テスト間でデータを共有しない
- グローバル状態に依存しない
- 順序に依存しない（順番をシャッフルしてもパスする）
- 各テストは前提条件を自分でセットアップする

### 6. 1テスト1観点

1つのテストでは1つの観点のみ検証する:

- ❌ `creates and updates and deletes user` （3観点を1テスト）
- ✅ それぞれを別テストに分割

複数のアサーションは OK だが、それらが同じ観点を検証している場合に限る。

## テスト構成パターン

### AAA パターン

各テストを Arrange / Act / Assert の3段階で書く:

```
Arrange: テストデータ・モックの準備
Act:     テスト対象の実行
Assert:  結果の検証
```

### 標準テストケース構成（CRUD 系の例）

```
- 正常系
  - 期待される動作をする
  - オプションフィールドありの場合
- 異常系
  - 必須フィールド欠落でエラー
  - 不正な形式でエラー
  - 権限不足でエラー
  - データ不存在でエラー
  - 他テナントのデータでエラー
- 境界値
  - 空配列 / 空文字 / null
  - 最大値 / 最小値
- 副作用
  - 関連データの更新が起きる
  - イベント発火 / ログ記録
```

## テスト対象の優先順

すべてをテストできない場合、以下の優先順で選ぶ:

1. **書き込み系**（create / update / delete） — データ破壊リスク
2. **権限制御系** — セキュリティリスク
3. **複雑なビジネスロジック** — バグ混入リスク
4. **境界処理** — エッジケースのバグリスク
5. **読み取り系**（list / get） — データ漏洩リスク

## 失敗時の切り分け

テスト失敗時は **原因がどちら側か** 必ず切り分ける:

- **テストのバグ**: テスト記述・モック・期待値が間違っている → テストを直す
- **実装のバグ**: 実装が仕様を満たしていない → 実装を直す（implementer に差し戻し）
- **仕様の曖昧さ**: 仕様が決まっていない → architect に確認

## カバレッジの考え方

カバレッジは目的ではなく結果。「100% を目指す」のではなく:

- ビジネスロジックの分岐が網羅されているか
- 境界値ケースが網羅されているか
- 異常系パスが網羅されているか

を基準にする。
