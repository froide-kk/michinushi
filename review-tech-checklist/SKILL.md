# Skill: review-tech-checklist

## 目的

技術的観点でコード・設計をレビューする際の universal なチェックリスト。
プロジェクト固有のチェック項目は `.claude/config/tech.yml` に追加する。

## チェックカテゴリ

### セキュリティ

最優先のチェック項目。1つでも MUST 該当があれば差し戻し。

- [ ] 認証チェック: 保護リソースに認証ガードがあるか
- [ ] 認可チェック: 操作者の権限が確認されているか
- [ ] テナント分離: マルチテナントの場合、他テナントデータへのアクセスが防がれているか
- [ ] 入力検証: 全ての外部入力（API パラメータ、フォーム入力）にバリデーションがあるか
- [ ] インジェクション対策: SQL / NoSQL / コマンド / XSS / パストラバーサルの対策
- [ ] 機密情報の取扱い: ログ・エラーメッセージ・レスポンスへの漏洩がないか
- [ ] CSRF 対策: 状態変更操作に CSRF トークンが効いているか
- [ ] レート制限: ログイン・パスワードリセット等にレート制限があるか

### 型安全

- [ ] `any` の使用が必要最小限か（不要なら具体型に置き換え）
- [ ] 型アサーションが安全か（`as` の濫用がないか）
- [ ] null / undefined の扱いが明示的か
- [ ] 関数のシグネチャが明確か（戻り値の型推論に過度に依存していないか）
- [ ] 型定義の一貫性（同じ概念に複数の型が乱立していないか）

### エラーハンドリング

- [ ] 想定されるエラーを catch しているか
- [ ] エラーメッセージに内部情報（スタックトレース・SQL・パス）が含まれていないか
- [ ] エラーログが適切なレベルで出力されているか
- [ ] エラー時のリソースリーク（接続・ファイル・トランザクション）がないか
- [ ] try / catch の粒度が適切か（広すぎない、狭すぎない）

### パフォーマンス

- [ ] N+1 クエリがないか
- [ ] 不要なループ・再計算がないか
- [ ] 大量データの取り扱い（ページネーション・ストリーミング）
- [ ] 重い処理が同期で実行されていないか
- [ ] キャッシュが必要な箇所で機能しているか

### 既存パターンとの整合性

- [ ] 命名規則がプロジェクトの既存パターンに合っているか
- [ ] ディレクトリ構造・ファイル配置が踏襲されているか
- [ ] エラーハンドリングのパターンが統一されているか
- [ ] インポート順序・スタイルが揃っているか

### データベース

- [ ] スキーマ変更にマイグレーションが付随しているか
- [ ] インデックスが適切か（よく使われる検索条件にインデックスがあるか）
- [ ] トランザクションの境界が適切か
- [ ] カスケード削除・更新が意図通りか

### CI / ビルド

- [ ] 型チェックが通る
- [ ] テストが通る
- [ ] lint 違反がない
- [ ] ビルドが通る
- [ ] 依存関係の追加が `package.json` 等に反映されている

## レビュー時の進め方

1. `git diff` で変更箇所を確認
2. 上記カテゴリを順に検証
3. 各指摘について以下を明示:
   - 重大度（MUST / SHOULD / NIT）
   - 何が問題か
   - どう直すべきか（具体的な修正案）
   - 関連ファイル・行

## 重大度の判定基準

- **MUST**: セキュリティリスク・バグ・型エラー・テスト失敗・要件未達
- **SHOULD**: 保守性低下・将来のリスク・既存パターンからの逸脱
- **NIT**: スタイル・命名・コメント不足など軽微な改善
