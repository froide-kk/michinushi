# Skill: review-biz-checklist

## 目的

業務的観点でコード・設計をレビューする際の universal なチェックリスト。
「技術的に正しい」だけでなく「業務として正しい」かを判断する。
プロジェクト固有のチェック項目は `.claude/config/biz.yml` に追加する。

## チェックカテゴリ

### 要件整合性

- [ ] 元の要件・ユースケースを満たしているか
- [ ] 設計書（`docs/` 配下）に書かれた仕様と実装が一致しているか
- [ ] 暗黙の要件（マルチテナント分離・監査ログ等）が実装されているか
- [ ] 受け入れ条件（acceptance criteria）が満たされているか

### ドメインモデルの妥当性

- [ ] エンティティ間の関係が業務実態に即しているか
- [ ] 必須属性・任意属性の区別が業務ルールと合っているか
- [ ] 計算ロジック（金額・数量・期間）が業務的に正しいか
- [ ] ID 体系・コード体系が業務で扱える形か

### 状態遷移の妥当性

- [ ] 全ての状態が網羅されているか（Empty / Loading / Error 含む）
- [ ] 状態間の遷移条件が業務ルールに従っているか
- [ ] 不正な遷移がブロックされているか
- [ ] 遷移時の副作用（通知・履歴記録）が漏れていないか

### ユーザー体験

- [ ] エラーメッセージがユーザーに分かる言葉で書かれているか
- [ ] 操作のフィードバック（成功・失敗・進行中）が適切か
- [ ] 操作ステップ数が適切か（不必要に多くないか）
- [ ] 復帰可能性: 誤操作からのリカバリ手段があるか
- [ ] アクセシビリティ: キーボード操作・スクリーンリーダーで使えるか

### データ整合性

- [ ] 同時編集時の挙動（楽観的ロック・最新優先）が業務として妥当か
- [ ] 削除時のカスケード/論理削除が業務要件に沿っているか
- [ ] 履歴・監査ログが必要な操作で記録されているか
- [ ] 集計値・派生値の更新タイミングが正しいか

### 権限モデル

- [ ] ロール定義が業務上の役割と一致しているか
- [ ] 権限の付与・剥奪のフローが業務手順と合っているか
- [ ] 「見えるが操作できない」「見えない」の使い分けが妥当か

### 業務フロー全体

- [ ] 一連の業務が完結するか（途中で止まる箇所がないか）
- [ ] 例外フロー（キャンセル・差し戻し・取り消し）が考慮されているか
- [ ] 関連する他機能との整合性（既存業務への影響）

## レビュー時の進め方

1. 変更対象の機能を特定
2. 関連する上流設計書（要件定義、ドメインモデル、業務フロー）を読み込む
3. 上記カテゴリを順に検証
4. 各指摘について以下を明示:
   - 重大度（MUST / SHOULD / NIT）
   - 何が問題か
   - 関連する要件・設計書
   - 改善案

## 重大度の判定基準

- **MUST**: 要件未達・業務破綻・データ不整合
- **SHOULD**: ユーザビリティ低下・将来の業務拡張への悪影響
- **NIT**: 用語の不統一・軽微な UX 改善

## tech-reviewer の修正後の追加チェック

技術観点で修正が入った場合、業務的な副作用を確認:

- エラーメッセージの変更でユーザーに影響はないか
- バリデーション追加で正常な業務操作がブロックされないか
- パフォーマンス改善で業務上の挙動が変わっていないか
