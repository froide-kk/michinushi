# Skill: impl-coding-conventions

## 目的

実装時に守るべき universal な原則をまとめる。
言語・フレームワーク固有の規約はプロジェクトの `.claude/config/tech.yml` に記述し、本 Skill はそれを補完する。

## 基本原則

### 1. 既存パターンを尊重する

新しいコードを書く前に、必ず周辺コードを読む:

- **命名規則**: ファイル名・関数名・変数名のパターン
- **ディレクトリ構造**: どこに何を置くか
- **エラーハンドリング**: try/catch の使い方、エラー型の慣習
- **インポート順序**: 標準ライブラリ → 外部 → 内部の順序やエイリアス
- **コメント**: どこまで書くか（過剰でも過少でもなく既存に揃える）

3つ以上の既存ファイルで同じパターンが使われていれば、それは「規約」とみなす。

### 2. 最小変更

要求された変更のみ行う:

- 周辺コードのリファクタは勝手にしない
- 「ついでに」の改善は別タスクとして報告
- フォーマッタ・linter による広範な変更は避ける（差分が読めなくなる）

### 3. 段階的なコミット可能性

各ステップで以下を満たす状態を保つ:

- ビルド・型チェックが通る
- 既存テストが落ちない
- import / export が整合している

途中の状態で破壊的変更が混ざらないよう、論理的に分割する。

### 4. 設計からの逸脱は報告

実装中に「この設計だとうまくいかない」と気づいた場合:

- 勝手に設計を変えない
- Conductor に逸脱の理由と提案を報告
- 承認を得てから実装方針を変更

### 5. 設計書の同期

実装と設計書がずれることを避ける:

- 実装変更に伴って設計書（YAML）も更新する場合は同じ作業単位で行う
- `doc-yaml-schema` Skill を参照

## ファイル変更の進め方

### 新規ファイル作成

1. 同種のファイルを1つ読み、フォーマットを把握
2. 必要に応じてファイルテンプレートを使用
3. 新規ファイル作成後、関連する index / export ファイルへの追加を忘れない

### 既存ファイル変更

1. ファイル全体を読んでから編集（部分読みで誤った前提を持たない）
2. 変更箇所の前後の文脈を保つ
3. 不要な変更（空白・改行・コメント）を含めない

## コミット粒度の目安

**1コミットの理想**:
- 単一の論理的変更
- メッセージで「なぜ」が説明できる粒度
- 後でレビューする際に追える単位

**避けるべきパターン**:
- 複数の独立した変更を1つにまとめる
- 自動生成ファイルと手動変更を混ぜる（分けてコミット）
- WIP（Work In Progress）状態のコミット（rebase で整理）

## 検証のチェックリスト

実装完了時に確認:

- [ ] 型チェックが通る
- [ ] 既存テストが落ちていない
- [ ] import / export の整合性
- [ ] 新規依存関係の追加（`package.json` 等）が完了
- [ ] 設計書との整合性
- [ ] 環境変数・設定ファイル変更の README 反映
